# গিটহাব এবং ভার্সেল (Vercel) CI/CD ওয়ার্কফ্লো গাইড

এই ডকুমেন্টে গিটহাবের ব্রাঞ্চিং স্ট্র্যাটেজি এবং একটি প্রজেক্ট ম্যানেজ করার জন্য প্রয়োজনীয় সমস্ত গিট কমান্ডগুলো বিস্তারিত আলোচনা করা হয়েছে।

## বেসিক কনসেপ্ট

### ব্রাঞ্চ (Branch) কী?
ব্রাঞ্চ হলো আপনার মেইন প্রজেক্টের একটি আলাদা স্পেস বা 'কপি'। যখন আপনি `main` থেকে `dev` ব্রাঞ্চ তৈরি করেন, তখন `main` ব্রাঞ্চের ঐ মুহূর্তের সমস্ত কোডের একটি হুবহু কপি `dev` ব্রাঞ্চে চলে আসে। এরপর আপনি `dev` ব্রাঞ্চে যাই পরিবর্তন করুন না কেন, তা `main` ব্রাঞ্চকে প্রভাবিত করবে না। 

### পুল রিকোয়েস্ট (Pull Request - PR) কী?
আপনি যখন `dev` ব্রাঞ্চে কাজ শেষ করে কোড গিটহাবে পুশ করেন, তখন সেই কোড সরাসরি `main` ব্রাঞ্চে যুক্ত হয় না। কোডটি `main`-এ যুক্ত করার জন্য গিটহাবের কাছে একটি "অনুমতি বা রিকোয়েস্ট" পাঠাতে হয়, একেই Pull Request (PR) বলে। PR ওপেন করলে আপনি দেখতে পারবেন কী কী কোড পরিবর্তন হয়েছে এবং Vercel ঐ PR-এর জন্য একটি প্রিভিউ লিংক তৈরি করে দেয়।

### মার্জ (Merge) কী?
PR চেক করার পর যখন আপনি নিশ্চিত হন যে সবকিছু ঠিক আছে, তখন "Merge" বাটনে ক্লিক করলে `dev` ব্রাঞ্চের নতুন কোডগুলো `main` ব্রাঞ্চের সাথে যুক্ত হয়ে যায়। একেই মার্জিং (Merging) বলে।

## প্রয়োজনীয় গিট কমান্ডসমূহ (Git Commands)

### ১. ব্রাঞ্চ ম্যানেজমেন্ট (Branch Management)

**ব্রাঞ্চ দেখা (Viewing Branches):**
```bash
# শুধু আপনার লোকাল কম্পিউটারের ব্রাঞ্চগুলো দেখতে
git branch

# গিটহাব (রিমোট) এবং লোকাল কম্পিউটারের সব ব্রাঞ্চ একসাথে দেখতে
git branch -a

# শুধু গিটহাবের (রিমোট) ব্রাঞ্চগুলো দেখতে
git branch -r
```

**ব্রাঞ্চ তৈরি ও রিমোটে পাঠানো (Creating & Pushing Branches):**
```bash
# লোকাল কম্পিউটারে নতুন একটি ব্রাঞ্চ তৈরি করা এবং সেই ব্রাঞ্চে সুইচ করা (যেমন: dev)
# (নোট: এটি শুধু আপনার লোকাল পিসিতেই ব্রাঞ্চটি তৈরি করবে, গিটহাবে নয়)
git checkout -b dev

# লোকাল কম্পিউটারে তৈরি করা নতুন ব্রাঞ্চটি গিটহাবে (রিমোটে) পাঠাতে এবং কানেক্ট করতে
# (এটি দিলে গিটহাবেও dev নামের একটি ব্রাঞ্চ তৈরি হয়ে যাবে)
git push -u origin dev
```

**ব্রাঞ্চ পরিবর্তন বা সুইচ করা (Switching Branches):**
```bash
# এক ব্রাঞ্চ থেকে অন্য ব্রাঞ্চে যেতে (যেমন: main ব্রাঞ্চে যেতে)
git checkout main
# বা, git switch main
```

**গিটহাবের (রিমোট) ব্রাঞ্চ লোকালে আনা (Accessing Remote Branch):**
```bash
# অন্য কেউ বা অন্য পিসি থেকে গিটহাবে নতুন ব্রাঞ্চ বানালে, প্রথমে সেটি লোকালে ট্র্যাকিংয়ের জন্য আনতে হবে
git fetch

# এরপর সেই নতুন ব্রাঞ্চে সুইচ করতে পারবেন
git checkout target_branch_name
```

**ব্রাঞ্চ ডিলিট করা (Deleting Branches):**
```bash
# লোকাল কম্পিউটার থেকে কোনো ব্রাঞ্চ ডিলিট করতে (যে ব্রাঞ্চ ডিলিট করবেন, আপনি সেই ব্রাঞ্চে থাকতে পারবেন না। অন্য ব্রাঞ্চে গিয়ে ডিলিট করতে হবে)
git branch -d branch_name

# যদি কোনো ব্রাঞ্চ মার্জ না হয়ে থাকে এবং জোরপূর্বক (Force) ডিলিট করতে চান
git branch -D branch_name

# গিটহাব (রিমোট) থেকে কোনো ব্রাঞ্চ ডিলিট করতে
git push origin --delete branch_name
```

### ২. কোড সেভ বা কমিট করা (Committing Code)
লোকাল কম্পিউটারে কাজ করার পর সেই আপডেটগুলো সেভ করার জন্য:

```bash
# প্রজেক্টের সমস্ত নতুন বা পরিবর্তিত ফাইলগুলো ট্র্যাকিংয়ের জন্য অ্যাড করা
git add .

# নির্দিষ্ট কোনো ফাইল অ্যাড করতে চাইলে (যেমন: index.js)
# git add index.js

# পরিবর্তনগুলোর একটি সুন্দর মেসেজ দিয়ে সেভ (Commit) করা
git commit -m "এখানে আপনার আপডেটের বিবরণ লিখুন"
```

### ৩. গিটহাবে কোড পাঠানো (Pushing Code)

```bash
# নতুন তৈরি করা ব্রাঞ্চ প্রথমবারের মতো গিটহাবে পাঠাতে (আগেই ব্রাঞ্চ ম্যানেজমেন্ট সেকশনে বলা হয়েছে)
git push -u origin branch_name

# পরবর্তীতে একই ব্রাঞ্চে কাজ করার পর কোড পাঠাতে শুধু নিচের কমান্ডটি দিলেই হবে
git push
```

### ৪. গিটহাব থেকে কোড লোকাল পিসিতে নিয়ে আসা (Pulling Code)
ধরা যাক, আপনি অন্য কম্পিউটার থেকে কোড পুশ করেছেন বা PR মার্জ করার পর গিটহাবের `main` ব্রাঞ্চ আপডেট হয়েছে। সেই আপডেট লোকাল পিসিতে আনতে:

```bash
# যে ব্রাঞ্চ আপডেট করতে চান, প্রথমে সেই ব্রাঞ্চে যেতে হবে (যেমন: main)
git checkout main

# গিটহাব থেকে সর্বশেষ আপডেট ডাউনলোড করে লোকাল পিসির সাথে সিঙ্ক করা
git pull origin main
```

### ৫. অন্যান্য প্রয়োজনীয় কমান্ড (Miscellaneous Commands)

```bash
# প্রজেক্টের বর্তমান অবস্থা দেখতে (কোনো ফাইল পরিবর্তন হয়েছে কি না বা সেভ করা বাকি আছে কি না)
git status

# পুরোনো কমিটের হিস্ট্রি দেখতে
git log

# নির্দিষ্ট কোনো ফাইলের পরিবর্তন আগের অবস্থায় ফিরিয়ে নিতে (Discard changes)
git checkout -- file_name.js

# পুরো প্রজেক্টের সব পরিবর্তন আনডু (Undo) করে সর্বশেষ কমিটের অবস্থায় ফিরে যেতে (সতর্কতা: এতে সেভ না করা সব কাজ মুছে যাবে)
git reset --hard
```

## Vercel-এর সাথে টেস্টিং ফ্লো

যেহেতু আপনার প্রজেক্টটি Vercel-এ হোস্ট করা, তাই পুরো টেস্টিং ফ্লো-টি নিচের মতো হবে:

১. **লোকাল কাজ:** লোকাল কম্পিউটারে `dev` ব্রাঞ্চে কাজ করুন।
২. **গিটহাবে পুশ:** কাজ শেষ হলে গিটহাবের `dev` ব্রাঞ্চে কোড পুশ করুন (`git push origin dev`)।
৩. **Vercel Preview:** পুশ করার সাথে সাথেই Vercel স্বয়ংক্রিয়ভাবে আপনার `dev` ব্রাঞ্চের জন্য একটি **Preview URL** তৈরি করবে (Vercel ড্যাশবোর্ডে বা গিটহাবের কমিট হিস্ট্রিতে এই লিংকটি পাবেন)। এই লিংকটি লাইভ সাইট থেকে আলাদা, তাই লাইভ সাইটের কোনো ক্ষতি হবে না।
৪. **টেস্টিং ও PR:** লিংকে গিয়ে চেক করুন। সব ঠিক থাকলে গিটহাবে গিয়ে `dev` থেকে `main` ব্রাঞ্চে একটি Pull Request (PR) ওপেন করুন।
৫. **লাইভ রিলিজ (Merge):** গিটহাবে PR মার্জ করে দিন। মার্জ হওয়ার সাথে সাথেই Vercel আপনার `main` ব্রাঞ্চের কোড নিয়ে লাইভ প্রোডাকশন সাইট আপডেট করে দেবে।
