🏢 Enterprise-Level Git & GitHub Workflow Guide

একটি সফল সফটওয়্যার প্রজেক্ট পরিচালনার সঠিক ব্রাঞ্চিং লাইফসাইকেল, টিম কোলাবোরেশন এবং রিলিজ ম্যানেজমেন্টের বিস্তারিত নির্দেশিকা

📋 গাইড ইনডেক্স

  1. ১. এন্টারপ্রাইজ ব্রাঞ্চিং স্ট্র্যাটেজি ও লাইফসাইকেল
  2. ২. অস্থায়ী বা সাময়িক ব্রাঞ্চের বিস্তারিত কার্যপ্রণালী
  3. ৩. প্রোডাকশন ও প্রিভিউ অটো-ডিপ্লয়মেন্ট ফ্লো
  4. ৪. টিম কোলাবোরেশন ও গিট কমান্ডস
  5. ৫. সঠিক উপায়ে Pull Request (PR) ও কোড রিভিউ
  6. ৬. কনফ্লিক্ট এড়ানোর কৌশল ও সমাধান
  7. ৭. প্রোডাকশন হটফিক্স (Hotfix) প্রসেস

🌳 ১. এন্টারপ্রাইজ ব্রাঞ্চিং স্ট্র্যাটেজি ও লাইফসাইকেল

এন্টারপ্রাইজ লেভেলে প্রজেক্টের কোডবেস সুরক্ষিত রাখতে দুই ধরনের ব্রাঞ্চ ক্যাটাগরি ব্যবহার করা হয়: Permanent (স্থায়ী) Branches এবং Temporary (অস্থায়ী/সাময়িক) Branches।

🔄 ব্রাঞ্চের প্রকারভেদ ও ক্যারেক্টারিস্টিকস:

ব্রাঞ্চ ক্যাটাগরি ব্রাঞ্চের নাম ব্যবহার লাইফসাইকেল
Permanent main (বা master) লাইভ প্রোডাকশন কোড। এই ব্রাঞ্চের কোড সরাসরি এন্ড-ইউজাররা দেখতে পান। অনন্তকাল (কখনো ডিলিট হয় না)
Permanent dev (বা develop) কোড ডেভেলপমেন্টের মূল ইন্টিগ্রেশন পয়েন্ট। এটি স্ট্যাজিং এনভায়রনমেন্ট হিসেবে কাজ করে। অনন্তকাল (কখনো ডিলিট হয় না)
Temporary feature/* যেকোনো নতুন ফিচার বা আইডিয়া ডেভেলপ করার জন্য আলাদা কাজের জায়গা। সাময়িক (মার্জ শেষে ডিলিট করা হয়)
Temporary bugfix/* ডেভেলপমেন্ট বা স্ট্যাজিং সার্ভারে পাওয়া বাগ ফিক্স করার জন্য। সাময়িক (মার্জ শেষে ডিলিট করা হয়)
Temporary hotfix/* সরাসরি লাইভ প্রোডাকশনে পাওয়া জটিল বাগ সাথে সাথে সমাধান করার জন্য। সাময়িক (রিলিজ শেষে সাথে সাথে ডিলিট করা হয়)

⌛ ২. অস্থায়ী বা সাময়িক ব্রাঞ্চের বিস্তারিত কার্যপ্রণালী (Why Temporary Branches?)

"ফিচার বা বাগ ব্রাঞ্চগুলো কি অস্থায়ী?" — হ্যাঁ, এগুলো ১০০% অস্থায়ী বা সাময়িক ব্রাঞ্চ। এদেরকে গিটফ্লোতে Short-lived branches বলা হয়।

মূল উদ্দেশ্য: Isolation (কোড আলাদা রাখা)
একটি প্রজেক্টে যখন একাধিক ডেভেলপার কাজ করেন, তখন প্রত্যেকে নিজের জন্য একটি করে আলাদা feature/* ব্রাঞ্চ তৈরি করেন। এতে একজনের আনফিনিশড (অর্ধেক করা) কোড অন্য ডেভেলপারের কাজে কোনো বাধা সৃষ্টি করে না। কাজ শেষ হয়ে গেলে কোডটি টেস্ট করে মূল ব্রাঞ্চে পাঠিয়ে দিয়ে অস্থায়ী ব্রাঞ্চটি মুছে ফেলা হয়।

🛠️ Feature Branches (feature/feat-name)

• তৈরি হয়: সর্বদা dev ব্রাঞ্চ থেকে।
• লাইফসাইকেল: কাজ শুরু থেকে শেষ পর্যন্ত। কাজ সম্পূর্ণ হলে GitHub-এ Pull Request এর মাধ্যমে dev-এ মার্জ হয়।
• মুছে ফেলা: মার্জ সম্পন্ন হওয়ার সাথে সাথে রিমোট ও লোকাল উভয় জায়গা থেকে ডিলিট করা হয়।

🐛 Bugfix Branches (bugfix/bug-name)

• তৈরি হয়: সাধারণত dev ব্রাঞ্চ থেকে।
• ব্যবহার: QA টেস্টিংয়ের সময় কোনো বাগ ধরা পড়লে তা ফিক্স করতে এই ব্রাঞ্চ ওপেন করা হয়।
• লাইফসাইকেল: বাগ ফিক্স হওয়া পর্যন্ত। ফিক্স হওয়ার পর dev-এ মার্জ করে ডিলিট করে দেওয়া হয়।

🔄 একটি অস্থায়ী ফিচার ব্রাঞ্চের পূর্ণাঙ্গ জীবনচক্র (Branch Lifecycle Timeline):

১. ক্রিয়েশন (Checkout): ডেভেলপার dev থেকে লেটেস্ট কোড নিয়ে নতুন ব্রাঞ্চ বানান: git checkout -b feature/login-page
২. কোড ডেভেলপমেন্ট: লোকাল ব্রাঞ্চে কাজ সম্পন্ন করে বারবার কমিট ও পুশ করা হয়।
৩. পুল রিকোয়েস্ট (PR): কাজ সম্পন্ন হলে গিটহাবে feature/login-page থেকে dev-এ মার্জ করার জন্য PR ওপেন করা হয়।
৪. রিভিউ ও মার্জ: টিম লিড কোড রিভিউ করে অ্যাপ্রুভ করলে কোডটি dev-এ মার্জ হয়ে যায়।
৫. ডিলিশন (Cleanup): মার্জ হওয়ার সাথে সাথে রিমোট গিটহাবের পেজ থেকেই ব্রাঞ্চ ডিলিট করার বাটন পাওয়া যায়। এরপর লোকালেও git branch -d feature/login-page দিয়ে এটি মুছে ফেলা হয়।

⚙️ ৩. প্রোডাকশন ও প্রিভিউ অটো-ডিপ্লয়মেন্ট ফ্লো

অটোমেশন লাইফসাইকেল: আধুনিক এন্টারপ্রাইজ প্রজেক্টে Vercel/Netlify বা GitHub Actions-এর মাধ্যমে CI/CD সেটআপ করা থাকে।
১
Dev Branch (Staging Preview): আপনি যখনই কোনো ফিচার বা ডেভ ব্রাঞ্চে পুশ করেন, Vercel তার জন্য একটি সাময়িক প্রিভিউ লিংক তৈরি করে। এর ফলে মূল লাইভ সাইটে কোনো পরিবর্তন না ঘটিয়েই ফিচারটি টেস্ট করা সম্ভব হয়।
২
Main Branch (Live Production): যখন সমস্ত ফিচার dev ব্রাঞ্চে একত্রিত হয়ে সম্পূর্ণ রেডি হয়ে যায়, তখন dev থেকে main-এ পুল রিকোয়েস্ট মার্জ করা হয়। Vercel তখন লাইভ প্রোডাকশন সাইট আপডেট করে দেয়।

💻 ৪. টিম কোলাবোরেশন ও গিট কমান্ডস

টিমের সাথে কাজ করার সময় কোনো ফিচার শুরু ও শেষ করার আদর্শ কমান্ড ফ্লো:

১. নতুন কাজ শুরু করা:

PowerShell
# ১. dev ব্রাঞ্চে যান git checkout dev # ২. অন্যের করা কোনো লেটেস্ট চেঞ্জ থাকলে তা টেনে আনুন git pull origin dev # ৩. নিজের জন্য একটি অস্থায়ী ফিচার ব্রাঞ্চ তৈরি করুন git checkout -b feature/my-cool-feature

২. কাজ শেষ করে রিমোটে পাঠানো ও ডিলিট করা:

PowerShell
# ৪. কোড কমিট ও পুশ করুন git add . git commit -m "feat: complete login design" git push -u origin feature/my-cool-feature # (GitHub-এ PR মার্জ সম্পন্ন করার পর আপনার পিসিতে এসে ব্রাঞ্চটি ডিলিট করুন) # ৫. dev ব্রাঞ্চে ফিরে যান git checkout dev git pull origin dev # ৬. কাজ শেষ হওয়া সাময়িক ফিচার ব্রাঞ্চটি ডিলিট করে দিন git branch -d feature/my-cool-feature

🤝 ৫. সঠিক উপায়ে Pull Request (PR) ও কোড রিভিউ

টিমওয়ার্কের সবচেয়ে গুরুত্বপূর্ণ অংশ হলো PR। সরাসরি মার্জ না করে PR-এর মাধ্যমে টিমমেটদের কাছ থেকে ফিডব্যাক নেওয়া উচিত:

  • Title ও Description: PR-এর টাইটেল পরিষ্কার রাখুন (যেমন: fix: oauth redirection issue)।
  • Small PRs: হাজার লাইনের একটি বড় PR না করে ছোট ছোট ফিচারের জন্য আলাদা PR করুন। এতে রিভিউ করতে সুবিধা হয়।
  • Self-Review: PR ক্রিয়েট করার পর নিজে একবার কোড পরিবর্তনগুলো (Diff) দেখে নিন কোথাও অপ্রয়োজনীয় কমেন্ট বা টেস্ট কোড বাদ রয়ে গেছে কিনা।

⚔️ ৬. কনফ্লিক্ট এড়ানোর কৌশল ও সমাধান

টিমের অন্য কেউ একই ফাইলে কাজ করলে কনফ্লিক্ট হতে পারে। এটি এড়ানোর সেরা উপায় হলো কাজ করার সময় প্রতিদিন অন্তত একবার মূল dev ব্রাঞ্চ থেকে আপডেট আপনার ফিচার ব্রাঞ্চে টেনে আনা:

PowerShell
# নিজের ফিচার ব্রাঞ্চে থাকা অবস্থায়: git fetch origin git merge origin/dev

যদি কোনো কনফ্লিক্ট আসে, তা সাথে সাথে VS Code এ দেখাবে। সেখানে কোড রিকনসাইল বা ফিক্স করে পুনরায় কমিট করে দিন।

🔥 ৭. প্রোডাকশন হটফিক্স (Hotfix) প্রসেস

জরুরি নিয়ম: প্রোডাকশনে লাইভ কোনো মেজর বাগ সাথে সাথে সমাধান করতে এই ফ্লো ব্যবহার করা হয়।
PowerShell
# ১. সরাসরি main ব্রাঞ্চ থেকে hotfix ব্রাঞ্চ তৈরি করুন git checkout main git pull origin main git checkout -b hotfix/critical-payment-error # ২. বাগ সমাধান করে কমিট ও পুশ করুন git add . git commit -m "fix: critical payment gateway timeout" git push -u origin hotfix/critical-payment-error # ৩. GitHub-এ PR মার্জ সম্পন্ন হয়ে যাওয়ার পর dev ব্রাঞ্চের সাথেও সিঙ্ক করুন: git checkout dev git pull origin dev git merge main git push origin dev
